This guide explains how to run CodeCompanion next to athenahealth without disrupting how athena
bills. It covers the compliance posture (PHI, BAAs, the `ALLOW_REAL_PHI` gate) first, then the
concrete API integration steps, and finishes with an honest map of what is already built versus
what remains. The goal is a fast, safe win: the office-manager cockpit and clinical intelligence
live on athena's data this quarter, with billing left to athena.

---

## 1. The integration model — and why billing stays off

athenahealth stays your **system of record and your biller**. CodeCompanion sits beside it as the
office-manager **intelligence layer**, with its own billing rails switched **off**. CodeCompanion
pulls clinical encounters and charges from athena into its canonical model and runs its cockpit,
clinical intelligence, and eligibility on top — it never submits a claim for an athena practice and
never writes back to athena billing.

> **Why billing is off — this is by design, not a limitation.**
> athenahealth locks claim submission to its own clearinghouse (athenaEDI / ECR). Third-party
> billing through any other clearinghouse is **not permitted** — it is how athena monetizes the
> platform. So CodeCompanion runs in **"Clinical only (EHR billing)"** mode: one click in
> **Admin → Module control** turns the whole Billing & RCM group off. That control shipped in this
> build precisely for this scenario.

**What you gain right now**

- **The cockpit.** Command Center, Follow-up Queue, and Dashboard — the office-manager command center over athena's day.
- **Clinical intelligence on athena encounters.** Coding suggestions, Care Gaps, Audit Shield, Practice Pulse, Schedule Risk, and Revenue Analytics.
- **Real-time eligibility.** 270/271 verification via Stedi — read-only, no clearinghouse enrollment, and it does not conflict with athena's billing.
- **Found-money analytics (Phase 2).** Diff athena's submitted-vs-paid to surface underpayments, denials, and undercoding — read-only; it never re-submits a claim.

**Data flow (one-way pull)**

```
athena API                          CodeCompanion
  /claims, /patients,    ───▶  canonical model  ───▶  modules
  /insurances, /providers      (EhrEncounter → Claim)    (cockpit, intelligence, eligibility)
  (encounters + charges)

  # PHI is pulled IN and normalized. Nothing is written back to athena.
```

## 2. HIPAA, PHI & BAAs (read before any real data)

CodeCompanion treats compliance as **architecture, not a checklist** bolted on at the end. The
platform must not process real patient data until the gate below is fully closed. Until then
everything runs on synthetic or de-identified test data — the same mock-first posture every adapter
already follows.

**The hard gate (from `COMPLIANCE.md`)**

- **BAAs signed** with every vendor in the PHI path — including the new one, athenahealth.
- **Audit logging live** on every PHI access (already built: an immutable, append-only audit log).
- **RLS enforced** on every tenant-scoped table (already built and CI-guarded).
- **De-id gate tests passing** for the corpus transform (already built).

> **The master switch: `ALLOW_REAL_PHI`.** It stays `false` until the four items above are true. It
> gates every real-PHI rail, including the live athena client — `createAthenaSource` throws unless
> it is set. The app runs mock-first (`ATHENA_USE_MOCK=true`) until you deliberately flip it. Adding
> athena puts one more vendor in the PHI path, so it needs its own BAA.

**Business Associate Agreements you need**

| Vendor | Role in the PHI path | BAA | Status |
|---|---|---|---|
| **athenahealth** | Source of clinical encounters + charges (PHI in) | Yes — new | Pending |
| **Supabase** | Postgres database + hosting (stores PHI at rest) | Yes | Pending |
| **Anthropic** | Language tasks only (appeal / narrative drafting) | Yes | Pending |
| **Stedi** | Clearinghouse for 270/271 eligibility (PHI to payers) | Yes | Pending |
| **Stripe** | Payment processing (if cards touch PHI-linked balances) | Yes\* | Pending |

\*Stripe needs a BAA only if card activity is tied to PHI. Confirm scope with counsel.

**athena-specific compliance notes**

- **Minimum necessary.** The adapter requests only the MDP service scope
  (`athena/service/Athenanet.MDP.*`) and pulls only the encounter, charge, insurance, and provider
  fields needed to build a claim — nothing more.
- **The adapter boundary contains PHI.** athena field names never leave `lib/adapters/athena`;
  everything normalizes into the canonical model, which stays tenant-isolated under RLS.
- **The de-id corpus gate is untouched.** athena data is individual-claim PHI and can never reach
  the behavioral corpus — that path is structurally blocked and tested.
- **Diligence before sign-off.** Review athena's marketplace data terms and security posture as part
  of the BAA. Note athena exposes structured charge/payment objects, **not** raw X12 837/835 files.

> **Not legal advice.** This is engineering guidance to keep the build compliant-by-construction. It
> is not a substitute for a HIPAA security risk assessment or counsel. Get a qualified review and
> proper BAA sign-off before any real PHI flows.

## 3. Prerequisites (athena-specific)

Supabase, Vercel, and GitHub are assumed already set up. What is specific to athena:

- **Developer account.** Register at developer.athenahealth.com.
- **A registered application.** Create an app to obtain a `client_id` and `client_secret`. Start in Preview (sandbox).
- **Your practice + department.** The athenaNet `practiceid` (the Preview sandbox practice is `195900`) and at least one `departmentid`.
- **The MDP scope.** Enable the `athena/service/Athenanet.MDP.*` scope on the app.

You will set six environment variables in Vercel — see the reference table in section 5.

## 4. API integration steps

### Step 1 — Register the app in Preview
At developer.athenahealth.com, create an application, record its `client_id` and `client_secret`,
and enable the MDP scope. **Preview data is synthetic**, so connecting to it is safe before any BAA
is signed.

### Step 2 — Identify practice + department
Use Preview `practiceid` `195900` to start. For production, use your real `practiceid` and
`departmentid`(s).

### Step 3 — Configure the environment
Set these in a Vercel **staging/preview** deployment first (full table in section 5):

```bash
ATHENA_CLIENT_ID=...                       # from the Preview app
ATHENA_CLIENT_SECRET=...
ATHENA_PRACTICE_ID=195900                  # Preview sandbox practice
ATHENA_BASE_URL=https://api.preview.platform.athenahealth.com
ATHENA_USE_MOCK=false                      # use the real client (against Preview)
```

> **Sandbox vs. PHI — flip `ALLOW_REAL_PHI` in staging only.** The live athena client is gated
> behind `ALLOW_REAL_PHI`. athena Preview is synthetic, so to exercise the live client against
> Preview you set `ALLOW_REAL_PHI=true` in a **non-production staging deploy only** — and keep every
> other real rail mock (`CLEARINGHOUSE=mock` or `STEDI_SANDBOX=true`, no live Stripe key). **Never**
> set `ALLOW_REAL_PHI=true` in production until the BAA gate in section 2 is fully closed.

### Step 4 — Validate OAuth + the encounter pull
The adapter performs an OAuth2 client-credentials exchange (token cached until ~60s before expiry),
then pulls a day of claims and joins each to its patient, insurance, and provider. In staging,
confirm the token issues and a pull returns joined bundles:

```ts
import { createAthenaSource, athenaConfigFromEnv, pullAthenaEncounters } from '@/lib/adapters/athena'

const source = createAthenaSource(athenaConfigFromEnv())
const encounters = await pullAthenaEncounters(source,
  { serviceDateFrom: '2026-06-01', serviceDateTo: '2026-06-01' })
```

### Step 5 — Verify the canonical mapping
`pullAthenaEncounters` yields canonical `EhrEncounter`s; `encounterToClaim(e, 'athena')` turns each
into a canonical `Claim`. Confirm CPT/HCPCS, ICD-10 pointers, charge cents, subscriber, and payer
populate correctly.

> **One crosswalk to wire: athena payer id → EDI payer id.** athena's `insurancepackageid` is
> athena-internal. For real-time eligibility through Stedi, resolve it to the clearinghouse/EDI payer
> id via the `PayerDirectory` at the submission edge. This is the one remaining adapter task — see
> section 6.

### Step 6 — Turn billing off, then choose your modules
In **Admin → Module control**, click **"Clinical only (EHR billing)"** to turn the entire Billing &
RCM group off. Then re-enable the read-only analytics you want running alongside athena:

| Modules | State | Why |
|---|---|---|
| Command Center, Follow-up Queue, Dashboard | **On** | The cockpit — your daily drivers over athena's day. |
| Coding, Care Gaps, Audit Shield, Practice Pulse, Schedule, Analytics | **On** | Clinical intelligence on athena encounters. Read-only. |
| EHR Pull | **On** | Pulls encounters from athena into the canonical model. |
| Eligibility | **On** | Real-time 270/271 via Stedi. Read-only; no enrollment. |
| Found Money, A/R & Denials | **On (Phase 2)** | Analytics on athena's submitted-vs-paid. Turn on once payments are wired (section 6). |
| Claims (RCM), Clearinghouse, Enrollments, Scrubber, Patient Billing, Patient Balances | **Off** | athena owns billing and the clearinghouse. Do not run these. |

### Step 7 — Go to production
When the BAA gate is closed: sign the athenahealth BAA (and the rest), point `ATHENA_BASE_URL` at
production, use your real `practiceid`, set `ATHENA_USE_MOCK=false` and `ALLOW_REAL_PHI=true`, apply
the Supabase migrations, and run the readiness check until green.

```bash
ATHENA_BASE_URL=https://api.platform.athenahealth.com   # production
ATHENA_PRACTICE_ID=<your real practiceid>
ATHENA_USE_MOCK=false
ALLOW_REAL_PHI=true            # only after BAAs + security review

# then:  npm run readiness     # env-aware go/no-go check
```

## 5. Reference — OAuth, endpoints, environment variables

**What the adapter calls**

```
POST {base}/oauth2/v1/token
   Authorization: Basic base64(client_id:client_secret)
   grant_type=client_credentials&scope=athena/service/Athenanet.MDP.*

GET  {base}/v1/{practiceid}/claims?servicedatefrom=MM/DD/YYYY&servicedateto=..
GET  {base}/v1/{practiceid}/patients/{patientid}
GET  {base}/v1/{practiceid}/patients/{patientid}/insurances
GET  {base}/v1/{practiceid}/providers/{providerid}

# base = Preview  https://api.preview.platform.athenahealth.com
#        Prod     https://api.platform.athenahealth.com
```

**Environment variables**

| Variable | Staging / Preview | Notes |
|---|---|---|
| `ATHENA_USE_MOCK` | `false` | `true` = synthetic mock (default). `false` = real client. |
| `ALLOW_REAL_PHI` | `true` (staging only) | Master gate. Production: only after BAAs + review. |
| `ATHENA_CLIENT_ID` | `<preview app id>` | OAuth client id from developer.athenahealth.com. |
| `ATHENA_CLIENT_SECRET` | `<preview secret>` | OAuth client secret. Server-side only. |
| `ATHENA_PRACTICE_ID` | `195900` | Preview practice; your real `practiceid` in prod. |
| `ATHENA_BASE_URL` | preview URL | Preview vs. production base (see above). |

## 6. What's built vs. what's next (honest scope)

**Built in this repo** (`lib/adapters/athena`)

- **OAuth2 client.** Client-credentials token exchange with caching; injectable transport (unit-tested without a network).
- **Encounter + charge pull.** `getEncounterBundles` joins claims to patient, insurance, and provider for a service-date range.
- **athena → canonical mapping.** `athenaBundleToEhrEncounter` → `EhrEncounter` → `encounterToClaim`. Money as integer cents; no athena type escapes the adapter.
- **Mock source + tests.** A synthetic `MockAthenaSource` and six passing tests covering OAuth, the join, and the mapping.
- **The billing-off control.** Admin module control + the "Clinical only (EHR billing)" preset, built for exactly this integration.

**Next, to complete the athena value loop**

1. **Wire EHR Pull to a sync.** Connect `pullAthenaEncounters` to the EHR Pull screen and/or a scheduled job (Inngest) so encounters land daily.
2. **Payer-id crosswalk.** Resolve athena `insurancepackageid` → EDI payer id at the eligibility edge so one-click eligibility works on athena patients.
3. **Phase 2 — pull payments.** Pull athena payment/remittance objects and map them to the canonical remittance to power the Found-Money diff (submitted vs. paid vs. contracted) on athena's own data. This is the high-ROI "found money" proof, and it is read-only.

Effort is small-to-medium and stays entirely behind the mock + `ALLOW_REAL_PHI` gates.

## 7. Recommended sequence (checklist)

1. Register an athena Preview app; capture `client_id` / `client_secret`; note `practiceid` `195900`.
2. Set the env vars in a Vercel staging deploy: `ATHENA_USE_MOCK=false`, `ALLOW_REAL_PHI=true` (staging only, every other rail mock).
3. Validate OAuth + the encounter pull + the canonical mapping against Preview (synthetic data).
4. In Admin, apply "Clinical only (EHR billing)"; confirm the sidebar reflects athena-mode.
5. Wire the EHR Pull sync and the payer-id crosswalk (section 6).
6. In parallel (business / legal): sign the athenahealth BAA and the remaining BAAs; book the HIPAA security review.
7. Provision production: real `practiceid`, prod base URL, `ALLOW_REAL_PHI=true`, apply migrations, run `npm run readiness` until green.
8. Phase 2: wire athena payments → Found-Money to prove recovered revenue on athena's data.
